# 故障排查指南

## 1. 概述

本文档提供了常见故障的排查方法和解决方案，帮助开发人员快速定位和解决问题。

## 2. 分布式锁问题

### 2.1 获取锁超时

**问题现象**：
```
获取锁失败: key=order:create:1001:2001
```

**可能原因**：
1. Redis连接异常
2. 锁未正确释放
3. 锁过期时间太短

**排查步骤**：

```bash
# 1. 检查Redis连接
redis-cli -h localhost -p 6379 ping

# 2. 检查锁是否存在
redis-cli -h localhost -p 6379 GET lock:order:create:1001:2001

# 3. 检查锁的TTL
redis-cli -h localhost -p 6379 TTL lock:order:create:1001:2001

# 4. 查看Redis慢日志
redis-cli SLOWLOG GET 10
```

**解决方案**：
1. 检查Redis连接配置
2. 确保锁在finally块中释放
3. 增加锁过期时间
4. 使用锁续期机制

### 2.2 死锁问题

**问题现象**：
多个线程相互等待对方释放锁

**排查步骤**：

```bash
# 查看当前所有锁
redis-cli KEYS "lock:*"

# 查看锁的持有者
redis-cli GET lock:order:create:1001:2001
```

**解决方案**：
1. 按固定顺序获取锁
2. 设置锁超时时间
3. 使用tryLock带超时

## 3. 限流问题

### 3.1 限流误触发

**问题现象**：
正常请求被限流，返回429

**排查步骤**：

```bash
# 查看限流计数
redis-cli GET rate_limit:order:create:192.168.1.1

# 查看限流配置
grep -r "RateLimit" src/
```

**解决方案**：
1. 调整限流阈值
2. 检查限流key是否正确
3. 区分正常流量和异常流量

### 3.2 限流不生效

**问题现象**：
限流配置不生效，大量请求通过

**排查步骤**：
1. 检查@RateLimit注解是否正确
2. 检查AOP切面是否生效
3. 检查Redis连接

**解决方案**：
1. 确保注解在Controller方法上
2. 检查Spring AOP配置
3. 验证Redis连接

## 4. 缓存问题

### 4.1 缓存穿透

**问题现象**：
大量请求直接打到数据库

**排查步骤**：

```bash
# 查看缓存命中率
redis-cli INFO stats | grep keyspace

# 查看空值缓存
redis-cli GET user:999999
```

**解决方案**：
1. 启用空值缓存
2. 使用布隆过滤器
3. 参数校验

### 4.2 缓存击穿

**问题现象**：
热点key过期时，大量请求同时查数据库

**排查步骤**：
1. 查看数据库慢查询
2. 查看缓存过期时间

**解决方案**：
1. 使用互斥锁
2. 使用逻辑过期
3. 热点数据永不过期

### 4.3 缓存雪崩

**问题现象**：
大量缓存同时过期，数据库压力骤增

**排查步骤**：
1. 查看缓存过期时间分布
2. 查看数据库连接数

**解决方案**：
1. 随机过期时间
2. 多级缓存
3. 缓存预热

## 5. 事务问题

### 5.1 事务失效

**问题现象**：
数据不一致，事务未回滚

**排查步骤**：
1. 检查方法是否为public
2. 检查是否自调用
3. 检查异常是否被捕获

**常见原因**：
```java
// 1. 非public方法
@Transactional
private void doSomething() { } // 事务失效

// 2. 自调用
public void methodA() {
    methodB(); // 事务失效
}

@Transactional
public void methodB() { }

// 3. 异常被捕获
@Transactional
public void doSomething() {
    try {
        // 业务逻辑
    } catch (Exception e) {
        // 异常被吞，事务不回滚
    }
}
```

**解决方案**：
1. 方法改为public
2. 注入自身代理
3. 不捕获异常或手动回滚

### 5.2 死锁问题

**问题现象**：
数据库死锁，事务超时

**排查步骤**：

```sql
-- 查看当前事务
SELECT * FROM information_schema.INNODB_TRX;

-- 查看锁等待
SELECT * FROM information_schema.INNODB_LOCK_WAITS;

-- 查看死锁日志
SHOW ENGINE INNODB STATUS;
```

**解决方案**：
1. 按固定顺序访问资源
2. 保持事务简短
3. 使用合理的索引

## 6. 消息队列问题

### 6.1 消息丢失

**问题现象**：
消息发送成功但未被消费

**排查步骤**：
1. 检查RabbitMQ管理界面
2. 查看消息是否在队列中
3. 检查消费者是否正常

**解决方案**：
1. 开启生产者确认
2. 消息持久化
3. 消费者手动确认

### 6.2 消息重复消费

**问题现象**：
同一条消息被消费多次

**排查步骤**：
1. 查看消费者日志
2. 检查消息确认机制

**解决方案**：
1. 使用幂等消费
2. 数据库唯一索引
3. Redis去重

### 6.3 消息积压

**问题现象**：
队列中消息数量持续增长

**排查步骤**：

```bash
# 查看队列消息数
rabbitmqctl list_queues name messages consumers

# 查看消费者状态
rabbitmqctl list_consumers
```

**解决方案**：
1. 增加消费者数量
2. 优化消费逻辑
3. 增加队列数量

## 7. 性能问题

### 7.1 响应时间长

**排查步骤**：
1. 查看应用日志
2. 查看数据库慢查询
3. 查看Redis慢日志
4. 使用链路追踪

**常见原因**：
- 数据库查询慢
- 缓存未命中
- 外部服务调用慢
- GC停顿

### 7.2 内存溢出

**排查步骤**：

```bash
# 查看JVM内存
jmap -heap <pid>

# 导出堆内存
jmap -dump:format=b,file=heap.hprof <pid>

# 分析堆内存
jhat heap.hprof
```

**解决方案**：
1. 增加JVM内存
2. 优化代码减少内存使用
3. 修复内存泄漏

### 7.3 CPU使用率高

**排查步骤**：

```bash
# 查看CPU使用率
top -p <pid>

# 查看线程CPU使用
top -Hp <pid>

# 导出线程栈
jstack <pid> > thread.txt
```

**解决方案**：
1. 优化算法复杂度
2. 减少死循环
3. 优化正则表达式
