# 最佳实践总结

## 1. 概述

本文档总结了企业级并发与安全开发中的最佳实践，帮助开发人员避免常见陷阱，编写高质量代码。

## 2. 分布式锁最佳实践

### 2.1 锁的使用原则

```java
// ✅ 正确示例
public void processOrder(Long orderId) {
    String lockKey = "order:" + orderId;
    String lockValue = redisLockService.tryLock(lockKey, 30);
    if (lockValue == null) {
        throw new BusinessException("获取锁失败");
    }
    try {
        // 业务逻辑
        doProcess(orderId);
    } finally {
        redisLockService.unlock(lockKey, lockValue);
    }
}

// ❌ 错误示例
public void processOrder(Long orderId) {
    String lockKey = "order:" + orderId;
    redisLockService.tryLock(lockKey, 30);
    // 业务逻辑
    doProcess(orderId);
    // 忘记释放锁
}
```

### 2.2 锁的粒度选择

| 场景 | 锁粒度 | 示例 |
|------|--------|------|
| 订单操作 | 订单ID | `lock:order:{orderId}` |
| 用户操作 | 用户ID | `lock:user:{userId}` |
| 库存扣减 | 商品ID | `lock:product:{productId}` |
| 全局操作 | 固定key | `lock:global:config` |

### 2.3 锁的超时设置

```java
// 短时间操作：5-10秒
redisLockService.tryLock(key, 5);

// 中等操作：30秒
redisLockService.tryLock(key, 30);

// 长时间操作：60秒 + 续期
redisLockService.tryLockWithRenewal(key, 60);
```

## 3. 限流最佳实践

### 3.1 限流策略选择

| 场景 | 算法 | 说明 |
|------|------|------|
| API接口 | 滑动窗口 | 平滑限流 |
| 用户操作 | 固定窗口 | 简单高效 |
| 突发流量 | 令牌桶 | 允许突发 |

### 3.2 限流配置示例

```java
// API接口限流
@RateLimit(limit = 100, windowMs = 1000, type = LimitType.METHOD)
@GetMapping("/api/products")
public ApiResult<List<Product>> getProducts() { }

// 用户操作限流
@RateLimit(limit = 10, windowMs = 1000, type = LimitType.USER)
@PostMapping("/api/orders")
public ApiResult<Order> createOrder() { }

// IP限流
@RateLimit(limit = 1000, windowMs = 60000, type = LimitType.IP)
@GetMapping("/api/public/data")
public ApiResult<Object> getPublicData() { }
```

### 3.3 限流响应处理

```java
// 前端处理
if (response.code === 429) {
    const retryAfter = response.headers['Retry-After'];
    showMessage(`请求过于频繁，请${retryAfter}秒后重试`);
}

// 后端处理
@ExceptionHandler(BusinessException.class)
public ApiResult<?> handleBusinessException(BusinessException e) {
    if (e.getCode() == ErrorCode.RATE_LIMIT_EXCEEDED.getCode()) {
        return ApiResult.fail(429, e.getMessage());
    }
    return ApiResult.fail(e.getCode(), e.getMessage());
}
```

## 4. 缓存最佳实践

### 4.1 缓存Key设计

```java
// ✅ 正确示例：有业务含义
"user:info:{userId}"
"product:detail:{productId}"
"order:list:{userId}:{page}"

// ❌ 错误示例：无业务含义
"cache:1"
"data:key123"
```

### 4.2 缓存过期时间

```java
// 热点数据：较短过期时间
cacheService.set(key, value, 5 * 60); // 5分钟

// 配置数据：较长过期时间
cacheService.set(key, value, 60 * 60); // 1小时

// 静态数据：超长过期时间
cacheService.set(key, value, 24 * 60 * 60); // 24小时
```

### 4.3 缓存更新策略

```java
// Cache-Aside模式（推荐）
public Product getProduct(Long id) {
    String key = "product:detail:" + id;
    
    // 1. 查缓存
    Product product = cacheService.get(key);
    if (product != null) {
        return product;
    }
    
    // 2. 查数据库
    product = productMapper.findById(id);
    if (product != null) {
        // 3. 写缓存
        cacheService.set(key, product, 300);
    }
    
    return product;
}

// 更新时删除缓存
@Transactional
public void updateProduct(Product product) {
    // 1. 更新数据库
    productMapper.update(product);
    
    // 2. 删除缓存
    cacheService.delete("product:detail:" + product.getId());
}
```

## 5. 事务最佳实践

### 5.1 事务注解配置

```java
// ✅ 推荐配置
@Transactional(
    propagation = Propagation.REQUIRED,
    isolation = Isolation.READ_COMMITTED,
    timeout = 30,
    rollbackFor = Exception.class
)
public void createOrder(OrderRequest request) { }

// ❌ 不推荐配置
@Transactional // 使用默认配置，可能不符合业务需求
public void createOrder(OrderRequest request) { }
```

### 5.2 事务范围控制

```java
// ✅ 正确示例：事务范围小
@Transactional
public void createOrder(OrderRequest request) {
    // 核心业务逻辑
    orderMapper.insert(order);
    stockService.deduct(productId, quantity);
}

// ❌ 错误示例：事务范围大
@Transactional
public void createOrder(OrderRequest request) {
    // 核心业务逻辑
    orderMapper.insert(order);
    stockService.deduct(productId, quantity);
    
    // 非核心业务（不应该在事务中）
    emailService.sendNotification(user.getEmail());
    logService.saveLog(operation);
}
```

### 5.3 避免长事务

```java
// ✅ 正确示例：拆分事务
public void processOrder(Long orderId) {
    // 1. 查询数据（事务外）
    Order order = orderMapper.findById(orderId);
    User user = userMapper.findById(order.getUserId());
    
    // 2. 核心业务（事务内）
    orderService.processOrder(order, user);
    
    // 3. 后续操作（事务外）
    notificationService.send(user, order);
}

// ❌ 错误示例：长事务
@Transactional
public void processOrder(Long orderId) {
    // 查询
    Order order = orderMapper.findById(orderId);
    User user = userMapper.findById(order.getUserId());
    
    // 业务处理
    orderService.processOrder(order, user);
    
    // 耗时操作（不应该在事务中）
    notificationService.send(user, order); // 可能很慢
    reportService.generateReport(order);   // 可能很慢
}
```

## 7. 异常处理最佳实践

### 7.1 异常分类

```java
// 业务异常：用户可感知
throw new BusinessException(ErrorCode.STOCK_NOT_ENOUGH, "库存不足");

// 系统异常：记录日志
throw new SystemException("数据库连接失败", e);

// 第三方异常：重试或降级
throw new ThirdPartyException("调用支付接口失败", e);
```

### 7.2 异常处理原则

```java
// ✅ 正确示例：不吞异常
public void process() {
    try {
        doSomething();
    } catch (BusinessException e) {
        throw e; // 业务异常向上抛
    } catch (Exception e) {
        log.error("处理异常", e);
        throw new SystemException("处理失败", e);
    }
}

// ❌ 错误示例：吞异常
public void process() {
    try {
        doSomething();
    } catch (Exception e) {
        log.error("处理异常", e);
        // 异常被吞，调用方不知道失败
    }
}
```


## 11. 性能优化最佳实践

### 11.1 批量操作

```java
// ✅ 正确示例：批量插入
@Transactional
public void batchInsert(List<Order> orders) {
    int batchSize = 100;
    for (int i = 0; i < orders.size(); i++) {
        entityManager.persist(orders.get(i));
        if (i % batchSize == 0) {
            entityManager.flush();
            entityManager.clear();
        }
    }
}

// ❌ 错误示例：逐条插入
public void insertOneByOne(List<Order> orders) {
    for (Order order : orders) {
        orderMapper.insert(order); // 每次都执行SQL
    }
}
```
